# Ambientika Cloud Integration — Technical Reference

How devices, the mobile app, the cloud, and the local proxy interact.
For the binary TCP protocol see [`PROTOCOL.md`](PROTOCOL.md).
For the REST API reference see [`CLOUD-API.md`](CLOUD-API.md).

## 1. Device Provisioning (BLE)

Provisioning happens once per device — it writes the cloud host and WiFi credentials
into the device over Bluetooth so it knows where to connect on power-up.

```
Mobile App                     Device (BLE)
    │                               │
    │  ← advertises encryptedDeviceInfo (MAC + auth blob, device-generated)
    │                               │
    │  ── write H_<cloud-host>:11000 ──→   (sets cloud TCP endpoint)
    │  ── write S_<wifi-ssid> ─────────→   (WiFi network)
    │  ── write P_<wifi-password> ─────→
    │                               │
    │                         [connects to WiFi]
    │                         [connects to <cloud-host>:11000]
```

The `encryptedDeviceInfo` is generated by the device and readable from a BLE
characteristic during pairing. The mobile app reads it and uses it to register
the device in the cloud (see Section 2).

**Proxy implication:** to route device traffic through the HA proxy, the cloud
host written during BLE provisioning must be the HA host IP instead of
`app.ambientika.eu`. This means **all devices must be re-provisioned** whenever
the target host changes.

See `src/scripts/ble-provisioning.ts` for the provisioning script — update
`CLOUD_HOST` to `<HA-host-IP>:11000` before running.

## 2. Cloud Device Registration

After reading `encryptedDeviceInfo` from the device over BLE, the mobile app
registers it in the cloud:

```
Mobile App                     Cloud REST API
    │                               │
    │  POST /House/add-device-room  │
    │  {                            │
    │    encryptedDeviceInfo,       │  (read from device in step 1)
    │    deviceName,                │
    │    roomName,                  │
    │    houseId                    │
    │  }  ─────────────────────────→│
    │                               │  [cloud stores device by serial number]
    │  ← { id, serialNumber } ──────│
```

There is **no API endpoint to add a device by serial number directly** — the
`encryptedDeviceInfo` from BLE is the only registration path. If a device is
physically replaced, the old cloud entry must be removed in the app and the new
device re-provisioned via BLE.

## 3. Normal Device Operation (TCP)

Once provisioned the device maintains a persistent TCP connection to the cloud host.

```
Device                         Cloud TCP :11000
    │                               │
    │  ── TCP connect ─────────────→│
    │  ── 18b firmware info ───────→│   (sent once on connect)
    │  ── 21b status ──────────────→│   (every ~30 seconds)
    │  ←─ 15/16b setup ─────────────│   (role/zone assignment)
    │  ←─ 13b mode command ──────────│   (operating mode change)
    │  ←─ 9b filter reset ───────────│
```

**Online status rule:** the cloud marks a device online only after it receives
both the firmware packet (18b) and a status packet (21b) on the **same persistent
TCP connection**. Closing the connection immediately marks the device offline.

## 4. Proxy Operation

The proxy (`RemoteSocketService`) maintains one outbound cloud connection per
local device IP. Each device's TCP stream is forwarded in both directions.

```
Device          HA Proxy :11000       Cloud TCP :11000
    │                  │                      │
    │  ── connect ────→│                      │
    │                  │── TCP connect ───────→│   (one per device)
    │  ── 18b ────────→│── 18b ──────────────→│
    │  ── 21b ────────→│── 21b ──────────────→│   [device online in app]
    │                  │←─ setup/commands ─────│
    │←─ setup/commands─│                      │
```

The cloud also opens an **inbound back-channel** to the proxy on port 11000
immediately after the proxy connects outbound. This connection is used to
send setup packets and commands to devices. The proxy routes these to the
correct device by matching the MAC in bytes 2–7 of the packet.

## 5. Zone and Role Configuration

Devices are grouped into zones within a house. Each zone has exactly one
**Master** and one or more **Slaves**. The Master broadcasts UDP zone-sync
packets; Slaves do not.

Zone and role assignments are pushed by the cloud as 15/16-byte setup packets:

```
Cloud REST                     Cloud TCP → Proxy → Device
    │                               │
    │  POST /Device/apply-config    │
    │  (full House object) ────────→│
    │                               │── 15b setup → each device
    │  ← 200 OK (or 406 timeout) ───│
```

After a role change a device that becomes a **Slave** will no longer broadcast
UDP. Its zone sensor in HA is updated when the proxy receives and emits the
setup packet (`DEVICE_SETUP_UPDATE` event).

Role values: `Master=0`, `SlaveEqualMaster=1`, `SlaveOppositeMaster=2`.
Zone index maps to UDP broadcast port: `45000 + zoneIndex`.

## 6. House and Zone Structure

```
House
 ├── Zone "Wohnen/Kind"   (id: 31755)
 │    ├── Room LivingRoom   → device: Wohnzimmer    (Master,            zone 0)
 │    └── Room ChildrenRoom → device: Kinderzimmer  (SlaveOppositeMaster, zone 0)
 │
 └── Zone "Büro/Schlafen" (id: 31756)
      ├── Room DinningRoom  → device: Büro Alex     (Master,            zone 1)
      └── Room Bedroom      → device: Schlafzimmer  (SlaveOppositeMaster, zone 1)

House "Einliegerwohnung"
 └── (no zones — flat device list)
      ├── Room LivingRoom   → device: Wohnzimmer    (SlaveOppositeMaster, zone 0)
      └── Room Kitchen      → device: Küche         (Master,            zone 0)
```

To reconfigure roles use `POST /Device/apply-config` with the full House object.
See [`CLOUD-API.md`](CLOUD-API.md) for the request shape.

**Warning:** `POST /Device/apply-config-force-unique` removes all zones and
flattens the house to a single system. Avoid using it.

## 7. Re-Provisioning Checklist

When changing the cloud host (e.g. pointing devices at a new HA instance):

1. Update `CLOUD_HOST` in `src/scripts/ble-provisioning.ts` to `<HA-host-IP>:11000`
2. BLE provision each device (run the script with device MAC)
3. For any new/replaced device: extend the script to **read** `encryptedDeviceInfo`
   from the device's BLE characteristic, then call `POST /House/add-device-room`
   to register it in the cloud before provisioning
4. Verify each device appears in the proxy log (`Device connected: <IP>`)
5. Verify cloud shows devices online (`GET /Device/house-devices-status`)

## 8. Known Limitations

| Issue | Detail |
|-------|--------|
| No delete-device API | Replaced devices must be removed via the mobile app |
| No add-by-serial API | `encryptedDeviceInfo` from BLE is the only registration path |
| `apply-config` 406 | Returns 406 if device doesn't acknowledge within ~60s (device offline or proxy down) |
| Slave zone sensor | Slaves don't broadcast UDP — zone is only known after a cloud setup packet is received |
| Re-provisioning required | Changing the TCP host requires BLE access to each device |
